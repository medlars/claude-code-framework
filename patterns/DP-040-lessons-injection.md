## What it is

Two-tier lesson injection: every session @imports the compact
`LESSONS-INDEX.md` (~1.1k tokens), but the full `LESSONS.md` (~4k
tokens, 314 lines, 101 lessons) is only read on demand via
`python3.14 ~/Projects/shared/lesson.py <id>`. The index regenerates
automatically when `LESSONS.md` is edited (PostToolUse hook
`lessons-index-regen.sh`).

## Rationale — Why We Adopted This Pattern

*The story of how this pattern came to exist and why it was chosen over alternatives.*

### The problem we kept hitting
LESSONS.md was 4,000+ tokens. `@import`-ing it into every session burned
context and cache. Most lessons weren't relevant to most sessions.

### What we tried first (and why it didn't work)
- **`@import` LESSONS.md** — 4k tokens per session, cache miss on every
  edit.
- **Forget about it** — lessons not present, mistakes re-made.

### The insight that unlocked the solution
**Lazy-load lessons by ID.** LESSONS-INDEX.md (~1.1k tokens, categorized)
is imported into every session. When a task matches a category, the
model fetches the full lesson on demand with `lesson.py <id>`. Index
auto-regenerates via a PostToolUse hook when LESSONS.md changes.

### Why this approach, not the obvious one
*Why not just keep LESSONS.md short?* Because lessons accumulate;
truncating loses institutional knowledge. The lazy-load pattern decouples
*injection cost* from *catalog size*.

### Evidence that it works
- Cache hit rate up, session start time down, lessons still accessible
  when relevant.

## Why we use it

Always-loading the full lessons file would eat 4k tokens of context
per session. Loading only the index (one-line summaries) gives
discoverability at minimal cost; the full text loads on the rare turn
when a specific lesson is relevant.

The deeper principle: **context is the constraint; lazy injection is
the solution.**

## How it works

**Index format** (auto-generated by `build-lessons-index.py`):
```markdown
- **001** [2026-05-25] When session analysis reveals specific gaps...
- **002** [2026-05-25] Memory without a hook is advisory only.
- **003** [2026-05-22] When a sys.path.insert block exists in src/...
...
```

**Lookup tool** (`lesson.py`):
```bash
python3.14 ~/Projects/shared/lesson.py 027
# → prints lesson 027 verbatim
python3.14 ~/Projects/shared/lesson.py AP-013
# → prints anti-pattern AP-013
python3.14 ~/Projects/shared/lesson.py B-011
# → prints behavioral rule B-011
```

**Injection points:**
- `LESSONS-INDEX.md` @imported in `~/Projects/shared/LESSONS-INDEX.md`
  via `~/Projects/CLAUDE.md` chain.
- Per-task hook `lessons-before-plan-or-code.sh` reminds to query a
  lesson if a task matches a known category.

**Regeneration:** PostToolUse hook fires `build-lessons-index.py` after
any LESSONS.md edit so the index never drifts from the source.

## Example

Task: "Fix a TestWatch phantom gap."

Without lazy injection: full 4k-token LESSONS.md loaded; relevant
lesson 010 buried in noise.

With lazy injection:
- Session starts with 1.1k-token index.
- Hook recognizes "TestWatch" + "phantom" → suggests:
  `python3.14 ~/Projects/shared/lesson.py 010`
- Lesson 010 loads (150 tokens) on demand.
- Total cost: 1.25k tokens vs 4k.

## Known failure modes / lessons learned

- LESSONS-INDEX.md regeneration must happen automatically; manual
  regen drift caused stale-index incidents (PostToolUse hook fixed
  this).
- Index entries >200 chars break Markdown link rendering; format
  discipline enforced.
- LESSONS 002: Memory without a hook is advisory — same applies to
  lessons; the injection hook is what makes them effective.
